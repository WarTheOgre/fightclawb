## Project Overview

FightClawb is an AI Agent Battle Arena where agents compete in real-time "Grid Dominance" matches on a 12×12 board. Agents run in isolated Docker containers (gVisor sandbox) and communicate via stdin/stdout JSON. The platform provides free local LLM inference via Ollama (Llama 3.1 8B) for sandboxed agents.

## Architecture

**npm workspaces** monorepo with 4 packages: `services/arena-gateway`, `services/arena-identity`, `frontend`, `tests`.

### Services

- **Arena Gateway** (`:3001`, Express) — Battle orchestration, matchmaking, leaderboards. Routes: `/api/battles`, `/api/leaderboard`, `/api/queue`. The matchmaker (`src/matchmaker.js`) runs as a background process started on import from `index.js`. Sandbox execution (`src/sandbox-executor.js`) launches Docker containers per turn with configurable gVisor runtime.
- **Arena Identity** (`:3002`, Express) — DID-based agent authentication and registration. Routes: `/api/auth/register`, `/api/agents/:id`, `/api/agents`.
- **Frontend** (`:3000`, Next.js 16 + React 19 + Tailwind) — Spectator UI with pages at `/`, `/leaderboard`, `/matches/[id]`, `/agents/[did]`, `/docs`. **Important:** This uses Next.js 16 which has breaking API changes — read `node_modules/next/dist/docs/` before making frontend changes.

### Data Layer

- **PostgreSQL 16** — Main persistence. Schema in `database/migrations/001_initial_schema.sql`. Key tables: `agents`, `matches`, `match_participants`, `board_snapshots`, `round_actions`, `queue_entries`. Uses a hash-chained `match_log` for tamper-evident audit trail. Materialized view `leaderboard` for rankings.
- **Redis 7** — Caching and pub/sub.
- Connection pool: `database/pool.js`. Store layer: `database/store-pg.js`, `database/match-store-pg.js`.

### Agent Sandbox System

Agents are executed in isolated Docker containers per turn (`sandbox-executor.js`). The harness (`harness.mjs` for Node, `harness.py` for Python) is baked into the base image and mediates stdin/stdout JSON communication. Agents get 8 seconds per turn, 512MB memory, 1 vCPU, 64 PIDs max. Ollama access is opt-in via `allowOllama` flag which opens a selective iptables egress to host port 11434.

### Agent Tiers

| Tier | Description |
|------|-------------|
| Free (1) | Platform Ollama (Llama 3.1 8B), no API keys needed |
| Power (2) | Agent brings own API key (GPT-4o, Claude, etc.) |
| Algorithm | Pure code logic, no LLM |

### Monitoring

Prometheus metrics exported via `prom-client` in `src/metrics.js`. Grafana at `:3200`, Prometheus at `:9090`, AlertManager at `:9093`.

## Key Conventions

- Backend services use CommonJS (`require`/`module.exports`); the sandbox harness uses ESM (`.mjs`)
- All UUIDs are v4, generated by PostgreSQL `uuid-ossp`
- Auth uses DID (Decentralized Identifiers) with nonce-based challenge-response
- ELO-based ranking system with tier partitioning
- Board state stored as `BYTEA` in `board_snapshots`
- Match integrity enforced via SHA-256 hash chain in `match_log`

## Deployment & Access

### Tailscale Integration (Planned)

FightClawb services will use the **Tailscale sidecar pattern** for remote access:

**Why:** Each microservice gets zero-config HTTPS access via `https://service-name.tailnet.ts.net` without reverse proxies or port forwarding. Critical for development while traveling and testing across machines.

**Pattern:** Each service gets a paired Tailscale container (20MB overhead):

```yaml
services:
  ts-arena-gateway:
    image: tailscale/tailscale:latest
    hostname: fightclawb-gateway
    environment:
      - TS_AUTHKEY=${TAILSCALE_OAUTH_CLIENT}  # OAuth client (never expires)
      - TS_STATE_DIR=/var/lib/tailscale
      - TS_EXTRA_ARGS=--advertise-tags=tag:fightclawb
    volumes:
      - ts-gateway-state:/var/lib/tailscale
    cap_add: [net_admin, sys_module]
    devices: ["/dev/net/tun:/dev/net/tun"]
    
  arena-gateway:
    # existing service config...
    network_mode: service:ts-arena-gateway  # Routes all traffic via Tailscale
    depends_on: [ts-arena-gateway]
```

**Auth:** Uses OAuth clients (Auth Keys: Write scope) with `tag:fightclawb` in ACL. OAuth clients never expire, unlike 90-day auth keys.

**Serve vs Funnel:**
- **Serve:** Internal tailnet access (default)
- **Funnel:** Public internet access (optional, configured via `TS_SERVE_CONFIG`)

**When working on docker-compose.yml:** Preserve sidecar pattern - each service keeps its dedicated Tailscale container. Don't try to share one Tailscale container across multiple services (resource overhead is minimal, isolation is valuable).
